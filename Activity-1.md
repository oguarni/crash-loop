## 2. Game Genre

**Primary genre:** 2D Infrastructure & Automation Puzzle
**Adjacent genres:** Programming Puzzle · DevOps Simulation · Terminal-Aesthetic Indie

### 2.1 Concept Statement

A single-player puzzle game in which the player operates a fictional cloud platform from a monospaced terminal. Each level presents a failing distributed system — an overloaded queue, a flapping service, a frozen rollout — and the player composes nodes (load balancers, caches, test runners, CI gates, Infrastructure-as-Code templates) within a CPU / memory / cycle budget until simulated traffic completes inside the level's error budget. Optional gold-tier scoring rewards solutions that minimize cost or maximize automated test coverage, mirroring the Zachtronics tradition of triple-axis optimization.

### 2.2 Design Pillars

1. **Backend over Bullets.** Gameplay is driven by data structures, queues, and resource flow. No physics, no twitch reflexes.
2. **Infrastructure as Levels.** Every level is a small distributed system the player must reason about end-to-end.
3. **Minimalist Terminal Aesthetic.** Monospace typography, CRT-tinted palette, ASCII-leaning art. Low art-asset cost, high systemic depth.
4. **Educational Pivot.** Real software engineering practice — Infrastructure as Code, automated testing, load balancing, CI/CD — becomes first-class game mechanic rather than a metaphor stretched onto unrelated gameplay.

### 2.3 Why This Genre Fits a Software-Engineering Team

- No 3D pipeline, no physics solver, no networked multiplayer at v1 — the scope fits a one-semester academic timeline.
- The core loop reduces to deterministic state machines and tuned numeric balance — strengths a software engineering team already has.
- Each level is a self-contained pull request: the project iterates the way the team's everyday code already iterates.
- Translating concepts the team is currently studying (CI/CD, IaC, test coverage, observability) into mechanics turns the course's own theory into the game's content.
